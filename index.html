Conceitos de Git & GitHub

Git × GitHub

  • Git: sistema de controle de versão distribuído que roda na sua máquina.
  • GitHub: plataforma de hospedagem de repositórios Git na nuvem (issues, PRs, wiki, CI, etc.).

Termos essenciais

  • Repositório (repo): pasta versionada que guarda histórico.
  • Commit: “foto” das alterações + mensagem descritiva.
  • Branch: linha de desenvolvimento (ex.: main, feature/x).
  • Remote: repositório hospedado (ex.: origin no GitHub).
  • Clone: cópia local de um repositório remoto.
  • Fork: cópia do repo na sua conta GitHub para colaborar sem acesso de escrita.
  • Pull Request (PR): proposta para mesclar sua branch em outra (revisão + CI).
  • Stash: guarda mudanças temporariamente sem commitar.
  • Tag/Release: marca um ponto do histórico (ex.: v1.0.0).

Comandos básicos

git init                      # inicia repo local
git clone <url>               # clona repo remoto

git status                    # estado atual
git add <arquivos>            # prepara para commit (stage)
git commit -m "mensagem"       # salva snapshot

git branch                    # lista branches
git switch -c feature/x       # cria e troca para nova branch

git remote -v                 # remotes configurados
git push -u origin feature/x  # envia branch para remoto

git pull --rebase origin main # atualiza sua branch

Fluxos de Git & GitHub

GitHub Flow (simples e moderno)

  1. Criar branch a partir de main.
  2. Fazer commits pequenos e claros.
  3. Push para o remoto.
  4. Abrir Pull Request (descrição, checklist, referência a issues).
  5. Passar em CI e code review.
  6. Squash & Merge na main, remover branch.
  7. Tag/Release opcional para versões.

Git Flow (projetos com releases)

  • main: versões estáveis (tags).
  • develop: integra features antes de ir para main.
  • feature/*, release/*, hotfix/* para organizar entregas.
# exemplo de ciclo
# 1) partir de develop
git switch develop
git switch -c feature/login
# 2) trabalhar/commitar/push
# 3) PR: feature/login -> develop
# 4) release/* -> testes
# 5) merge em main + tag
Dica: padronize mensagens de commit, template de PR e convenções de nome de branch para agilizar o review.

Como Resolver Conflitos

Quando acontece? Ao mesclar histórico diferente que altera as mesmas linhas dos mesmos arquivos (ex.: merge, rebase, cherry-pick).

Passo a passo (merge)

  1. Traga a outra branch: git merge main ou git pull --rebase origin main.
  2. Abra os arquivos marcados com <<<<<<< e escolha/edite o conteúdo final.
  3. Marque como resolvido e faça commit: git add . → git commit.
  4. Finalize o merge/rebase e envie: git push.

Passo a passo (rebase)

git switch feature/x
git fetch origin
git rebase origin/main
# resolva conflitos arquivo a arquivo
# edite, depois:
git add <arquivo>
git rebase --continue
# para abortar:
# git rebase --abort

Receita rápida por situação

  • Conflito ao dar pull: git pull --rebase → resolver → git rebase --continue.
  • Conflito em PR: atualize sua branch local com a base, resolva e faça push.
  • Desfazer merge errado: git reset --hard <hash-anterior> (use com cuidado) ou git revert -m 1 <hash-do-merge>.
Evite conflitos: sincronize com frequência, faça PRs curtos e edite arquivos diferentes quando possível.
